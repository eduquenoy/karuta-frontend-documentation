Les métadonnées d'un élément Karuta servent à deux choses : lui attribuer un identifiant unique et spécifier son comportement.

L'identifiant unique de l'élément se nomme le Tag sémantique et permet de référencer cet élément à partir d'autres éléments. Il est de bonne pratique d'attribuer un identifiant unique à chaque élément ajouté à un portfolio.

On spécifie ensuite le comportement de chacun de ces éléments. Par exemple, dans le cas d'éléments de structure (Section, Page, Sous-section), on peut définir des menus qui permettent l'ajout d'éléments supplémentaires par le designer ou par l'utilisateur final. Dans le cas des éléments de ressource (éléments qui contiennent les données) on peut associer des rôles à chaque droit, définissant ainsi qui peut faire quoi sur l'élément.

C'est aussi dans les métadonnées qu'on spécifie quels sont les éléments qui vont peupler une boîte de choix et quels utilisateurs pourront partager leur portfolio avec une tierce partie (et sous quelles conditions de sécurité).

On accède aux métadonnées d'un élément via la boîte d'édition de celui-ci (bouton crayon).

Tag sémantique

Le tag sémantique d'un élément est son identifiant et devrait exprimer sa sémantique. Le designer d'un portfolio devrait, pour chaque élément qu'il ajoute à un portfolio, remplacer le tag sémantique par défaut par quelque chose qui exprime la sémantique de l'élément.

C'est par le tag sémantique qu'on recherche des éléments dans un portfolio. Il est utilisé dans la métadonnée Recherche (voir ci-dessous) ainsi que dans les scripts de rapport et de batch.

ATTENTION : La recherche de tag sémantique se fait par inclusion. Par exemple, si on recherche les éléments avec le tag sémantique competence, les éléments avec le tag sémantique sous-competence seront également trouvés, car le mot competence est inclus dans le mot sous-competence.

Code

Le plus souvent attribué aux éléments Item et Sous-section pour ensuite être utilisé dans les listes déroulantes (éléments Get_Resource et Get_Get_Resource) et les boîtes de choix multiples (fonctions Get_Multiple et Import_Multiple). C'est lui qui permet d'identifier la sélection de l'utilisateur.

ATTENTION : Le code d'un élément de portfolio doit être unique à l'intérieur de ce portfolio.

Par défaut le code et le libellé de l'élément apparaissent dans la liste déroulante et seul le libellé de l'option sélectionnée par l'utilisateur est ensuite afffiché dans l'élément Get_Resource (ou Get_Get_Resource). Ce comportement peut être modifié par l'ajout d'un ou plusieurs des symboles suivants au code de l'élément :

  • @ rendra le code non visible dans la liste déroulante
  • # rendra le code visible dans l'élément Get_Resource
  • & rendra la valeur associée à l'élément visible dans l'élément Get_Resource (seul les éléments de type Item ont un champ valeur)
  • % rendra le libellé non visible dans l'élément Get_Resource
  • $ rendra le libellé non visible dans la liste déroulante
  • ! rendra le choix non sélectionnable dans la boîte de choix multiple (utile pour créer des titres de section dans la boîte de choix)

Nœud multilingue

Spécifie si le libellé du nœud prend une valeur différente pour chaque langue dans le cas où le portfolio est multilingue. Si le nœud n'est pas multilingue, il y aura une seule valeur pour toutes les langues et cette valeur sera stockée dans la langue par défaut définie dans le fichier de configuration _init.js.

REMARQUE : Le code n'est jamais multilingue. Il a la même valeur quelle que soit la langue active.

Ressource multilingue

Éléments Ressource uniquement. Spécifie si les données entrées dans l'élément Ressource prennent des valeurs différentes pour chaque langue dans le cas où le portfolio est multilingue. Si la Ressource n'est pas multilingue, il y aura une seule valeur pour toutes les langues et cette valeur sera stockée dans la langue par défaut définie dans le fichier de configuration _init.js

Par exemple, un nom ou une date de naissance ne sont habituellement pas dépendants de la langue, alors qu'un texte qui donne des instructions est définitivement différent dans chaque langue.

Encrypter la ressource

Spécifie si la ressource doit être encryptée. Dans le cas positif, à l'ouverture du portfolio, une phrase-clé est demandée à l'utilisateur. Cette phrase-clé sert à encoder/décoder les données entrées dans la Ressource. Cette phrase-clé n'est pas stockée dans Karuta et doit donc être mémorisée par l'utilisateur, car elle lui est demandée à chaque ouverture du portfolio.

Voir

Liste des rôles, séparés par un espace, pouvant voir l'élément. Exemple : etudiant professeur.

Par défaut le droit de voir est attribué au rôle all, qui englobe tous les rôles. Il s'agit de l'unique droit à avoir une valeur par défaut.

Supprimer

Liste des rôles, séparés par un espace, pouvant supprimer l'élément.

Modifier ressource

Éléments Ressource uniquement. Liste des rôles, séparés par un espace, pouvant éditer la Ressource (y enregistrer des données).

Ressource sans crayon

Éléments Ressource uniquement. N'affiche pas le bouton crayon aux rôles ayant le droit d'éditer la Ressource. Utilisé pour les imports de composantes via la fonction de Menu Import_Multiple (voir section FONCTIONS MENU pour plus de détails).

Commenter

Liste des rôles, séparés par un espace, pouvant commenter un élément Ressource. Ce droit ajoute un éditeur de texte au bas de la boîte d'édition de la Ressource; le commentaire est daté et signé du nom de l'utilisateur qui l'a écrit. Pour pouvoir commenter un élément, l'utilisateur doit également avoir le droit de modifier la Ressource.

ASTUCE : Le designer peut s'attribuer le droit de commenter un élément de structure. Cela lui permet d'ajouter l'équivalent d'un sous-titre au libellé de l'élément.

Soumettre

Liste des rôles, séparés par un espace, pouvant soumettre l'élément. Ce droit ajoute un bouton Soumettre. Lorsque l'utilisateur soumet un élément, ses droits d'édition et/ou de suppression sont retirés pour l'élément et tous ses enfants.

Recherche

Le champ "recherche" est utilisé par :

  • les listes déroulantes (éléments Get_Resource et Get_Get_Resource)
  • les menus aux utilisateurs sur les éléments de structure (éléments Section, Page et Sous-section)
  • les listes de choix multiple implémentées à travers des menus (menu Get_Multiple, menu Get_Get_Multiple et menu Import_Multiple)

Format de la recherche pour un Get_ (élément Get_Resource ou menu Get_Multiple)

code_portfolio ou self.tag_sémantique.label ou fileid

  • code_portfolio : Code du portfolio référentiel où chercher le tag sémantique spécifié. Le format est 'code du projet.code du portfolio référentiel'. Si 'code du projet.' n'est pas présent, la recherche du portfolio référentiel se fait dans le projet courant (projet qui contient le Get_).
  • tag_sémantique : Tag sémantique des élements à chercher.
  • label : Mot réservé. Lorsque l'élément cherché est un Item ou une Sous-structure.

OPTION self : Peut remplacer code_portfolio. Dans ce cas la recherche se fait à l'intérieur du portfolio courant (portfolio qui contient l'élément Get_).

OPTION text : L'élément cherché doit être un Texte (court ou long). On remplace alors label par text. Dans ce cas, c'est le texte et non le libellé qui sera affiché.

OPTION fileid : L'élément cherché doit être une Image. On remplace alors label par fileid. Dans ce cas, l'élément Image doit être déclaré à même le portfolio où se trouve l'élément Get_ (et non pas dans un portfolio Référentiel) → donc à utiliser avec l'option self.

OPTION resource : L'élément cherché peut être un Item. On remplace alors label par resource. Dans ce cas, l'élément Item doit être déclaré à même le portfolio où se trouve l'élément Get_ (et non pas dans un portfolio Référentiel). L'effet sera contrairement à label, la mise à jour du get_resource si l'item correspondant sélectionné est modifié. → donc à utiliser avec l'option self.

Format de la recherche pour un Get_Get_ (élément Get_Get_Resource ou menu Get_Get_Multiple)

code_portfolio.position.tag_sémantique1. tag_sémantique2.label

  • code_portfolio : Code du portfolio référentiel où faire la recherche.
  • position : Indique où chercher dans le portfolio courant le code de l'élément parent qui va permettre de restreindre la recherche dans le référentiel. Prend une des valeurs suivantes : child, sibling,parent, parent.parent, parent.parent.parent
  • tag_sémantique1 : Tag sémantique du parent (restreint la recherche aux enfants de ce parent).
  • tag_sémantique2 : Tag sémantique des éléments à chercher dans le référentiel.
  • label : Mot réservé.

Partager

La fonction Partage par l'utilisateur permet à l'utilisateur d'envoyer un courriel à un tiers, avec un hyperlien qui partage, en tout ou en partie, son portfolio avec ce tiers.

Format

rôle
ou
rôle,rôle destinataire,rôles et/ou courriels,niveau (0-4),durée de vie du lien (en heures),libellé du bouton@fr, condition (optionel)

  • rôle Rôle de l'utilisateur qui peut faire le partage (qui aura le bouton pour partager).
  • rôle destinataire Rôle qu'aura le destinataire dans le portfolio.
  • rôles et/ou courriels Rôles et/ou courriels des destinataires. Si c'est un rôle, toutes les personnes ayant ce rôle dans le portfolio seront destinataires d'un courriel.
  • niveau (0-4) Le niveau peut prendre les valeurs de 0 à 4 (voir ci-dessous).
  • libellé du bouton@fr Libellé qui apparaîtra à l'utilisateur qui peut faire le partage.
  • condition Condition pour que le bouton de partage soit visible (qu'il existe). Exemple : submitted=='Y'
Niveau 0 – utilisateur Karuta

L’utilisateur (le destinataire du courriel) possède un compte Karuta et le portfolio lui est déjà partagé, dans le bon rôle. Son mot de passe lui sera demandé si il n'est pas déjà logué dans Karuta.

Niveau 1

L’utilisateur possède un compte Karuta et le portfolio lui est déjà partagé, dans le bon rôle. Son mot de passe ne lui sera pas demandé (si l'utilisateur n'est pas déjà logué dans Karuta, il obtiendra un login temporaire). Par mesure de sécurité, il devrait y avoir un bouton Soumettre sur les éléments que le visiteur peut modifier.

Niveau 2

L’utilisateur possède un compte Karuta mais le portfolio ne lui est pas partagé. Le système partage le portfolio avec l’utilisateur dans le bon rôle. (Si l'utilisateur n'est pas déjà logué, il obtiendra un login temporaire.) Par mesure de sécurité, il devrait y avoir un bouton Soumettre sur les éléments que le visiteur peut modifier.

Niveau 3

Si l’utilisateur n'a pas de compte dans Karuta, un compte lui sera créé. Le système partage le portfolio avec l'utilisateur dans le bon rôle. (Si l'utilisateur n'est pas déjà logué, il obtiendra un login temporaire.) Par mesure de sécurité, il devrait y avoir un bouton Soumettre sur les éléments que le visiteur peut modifier.

Niveau 4 – Lien public

Si l'utilisateur n'est pas logué dans Karuta, il obtient un login temporaire. Il a automatiquement (logué ou non) le rôle all (donc il peut voir mais ne peut rien modifier).

Pour une documentation complète et des exemples d'utilisation consulter Doc Partage par l'utilisateur.

Menus

Le champ Menus permet de présenter un menu déroulant de choix de composantes à importer. Le menu peut comprendre plusieurs éléments. Dans le cas où pour un rôle, il n'y a qu'un seul élément alors seul cet élément sera affiché. Si il ya plusieurs éléments pour un rôle, il est possible de contrôler le libellé du menu ('Menu' par défaut) en précisant le libellé et la langue. Les différents éléments de menu sont séparés par un ; (point-virgule). ATTENTION : il n'y a pas de point-virgule final.

Format

[libellé_menu@fr;]code_portfolio,tag sémantique,libellé@fr,rôles[; autres éléments]

  • libellé_menu@fr : remplacera le mot Menu par défaut si il ya plus d'un élément dans le menu. OPTIONNEL
  • code_portfolio : Code du portfolio duquel on importe la composante.
  • libellé@fr : Libellé de l'élément de menu.
  • rôles : Rôle(s) voyant l'élément de menu.
